# Данные в облаке (Supabase) и миграции

Цель: **ничего важного не должно жить только локально**. Схема и структура данных, которые нужны в продакшене и в облаке, описываются в репозитории и применяются к проекту Supabase через миграции. Так данные не «теряются» при смене машины, переустановке SQLite или деплое.

## Принципы

1. **Supabase (Postgres) — целевой источник правды для облака.** Любое изменение схемы (таблицы, колонки, индексы, ограничения, RLS, функции, триггеры, сиды структуры) должно иметь отражение в SQL-миграции в этом репозитории.
2. **Миграция в Git в том же изменении, что и код**, который от неё зависит (один PR / одна логическая серия коммитов). Не откладывать «потом допишем SQL».
3. **Не править продакшен руками** без зеркала в `supabase/migrations/` — иначе облако разъедется с репозиторием и с другими окружениями.
4. Локальный SQLite (если приложение ещё пишет в него) **не заменяет** облако: для долговечности и бэкапов критичные сущности должны быть выведены в Supabase и/или синхронизированы по согласованной схеме; новые фичи проектируются с учётом миграций под Postgres.

## Когда обязательно добавлять миграцию

Создайте новый файл в `supabase/migrations/`, если вы:

- добавили или переименовали **таблицу**;
- добавили, удалили или изменили тип **колонки**;
- добавили **индекс**, **внешний ключ**, **уникальное ограничение**;
- изменили **Row Level Security**, политики, роли;
- добавили **функцию или триггер** в БД;
- меняете структуру данных, без которой новый код на сервере **не сможет** работать в облаке.

Если изменение только в UI и **не трогает БД** — миграция не нужна.

## Как назвать и оформить файл

- Каталог: `supabase/migrations/`.
- **Нумерация (v2 и далее):** `001_kratkoe_opisanie.sql`, `002_...`, `003_...` — трёхзначный порядковый номер, подчёркивание, описание латиницей. Реестр: `supabase/migrations/README.md`.
- **Legacy (v1):** файлы `YYYYMMDDHHMMSS_...` — не переименовывать после применения на проде.
- SQL по возможности **идемпотентный**: `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, аккуратные `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` (если версия Postgres поддерживает), явные комментарии к неочевидным шагам.
- Язык комментариев в файле — русский или английский, но единообразно внутри одной миграции.

## Порядок работы (чеклист для разработчика и ИИ)

1. Спроектировать изменение схемы (таблица/колонки/индексы).
2. Создать файл миграции `supabase/migrations/NNN_opisanie.sql` (следующий номер — в `supabase/migrations/README.md`).
3. Проверить SQL в панели Supabase (SQL Editor) или через CLI на копии проекта.
4. Внести изменения в код приложения (API, `lib/*`), согласованные со схемой.
5. Закоммитить миграцию и код вместе; в описании PR/коммита указать, что затронута БД.
6. После мержа — применить миграции к целевому проекту Supabase (CI, `supabase db push`, или вручную по процессу команды — как принято у вас).

## Где смотреть пример

Актуальный пример v2: `supabase/migrations/001_v2_core_schema.sql`. Legacy v1: `20260512180000_team_tracker_pm_board_history.sql`.

## Напоминание для ИИ-агентов

При любой задаче, которая затрагивает **хранение данных**, **новые сущности** или **изменение существующей модели данных**, сначала или параллельно спланировать миграцию Supabase и добавить файл в `supabase/migrations/`. Не оставлять схему только в локальном SQLite или только «в голове» — облако и репозиторий должны оставаться согласованными.
